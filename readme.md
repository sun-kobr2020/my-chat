У меня есть веб-мессенджер (SPA) с комнатами, развернутый на GitHub Pages
и собранный на Vite (JS, HTML5, CSS3).

### Текущая гибридная архитектура проекта:
1. База данных и Real-time (Firebase v9+ Firestore): Текст сообщений, имена
   комнат и метаданные синхронизируются через Firebase в реальном времени
   (onSnapshot).
2. Собственное self-hosted хранилище (Node.js + Express + Multer): Все
   медиафайлы и документы загружаются и хранятся на моем домашнем сервере
   (ПК с 64 ГБ RAM), а не в Firebase Storage. Бэкенд проброшен в интернет
   через бесплатный туннель localtunnel (команда запуска:
   `npx localtunnel --port 5000 --print-requests`).

### Структура и файлы проекта:
1. index.html — Двухпанельный интерфейс. Внизу панели чата находится блок
   '.input-panel'. Прямо над строкой ввода '.message-row' встроен
   служебный блок индикации файлов `#file-preview-zone` с кнопкой отмены
   `#cancel-file-btn`.
2. style.css — Стили сетки чата, комнат и адаптивная верстка.
3. app.js — Главный скрипт (модульный SDK Firebase). Обрабатывает логику
   комнат, слушает Firestore, управляет отправкой и мультизагрузкой файлов
   через fetch с заголовком 'bypass-tunnel-reminder: true' (для обхода
   стартовой страницы localtunnel).
4. server.js — Бэкенд-сервер на Node.js (работает локально на порту 5000),
   принимающий файлы от фронтенда. Имеет двойной CORS-блок (cors() +
   ручные заголовки), включая обработку preflight OPTIONS-запросов и
   заголовка Bypass-Tunnel-Reminder.
5. .env — Содержит ключи VITE_FIREBASE_* и переменную VITE_FILE_SERVER_URL
   (в режиме разработки указывает на http://localhost:5000, в продакшене —
   на адрес туннеля localtunnel, вида https://*.loca.lt).
   Папка /uploads/ скрыта в .gitignore.

### Логика работы с файлами и сообщениями:
- Фронтенд поддерживает мультизагрузку (multiple) любых типов файлов через
  функцию handleFileUpload(e). Файлы предварительно копируются в массив
  через Array.from(e.target.files), а очистка input.value происходит ТОЛЬКО
  в блоке finally (после полного завершения загрузки), что устранило баг с
  обнулением FileList в браузере.
- Файлы отправляются поочерёдно в цикле через FormData (поле 'chatFile')
  на бэкенд, который динамически сортирует их на жестком диске по папкам:
  /uploads/images/, /uploads/videos/ или /uploads/documents/ на основе
  MIME-типов.
- Бэкенд возвращает объект {fileUrl, fileType} для каждого файла. Фронт
  собирает массив attachedFilesList. Серая плашка превью динамически
  меняет цвета и статусы («⏳ Отправка X из Y...» → «✅ Файлы на сервере!
  Готово: N шт.» → «🚀 Отправка в чат...»). В случае сбоя выводится
  ошибка красным.
- Функция sendMessage() упаковывает текст и массив файлов в один объект
  и отправляет в Firestore по пути: rooms / {currentRoomId} / messages.
  После публикации интерфейс и массив attachedFilesList полностью
  очищаются.
- Рендеринг внутри onSnapshot автоматически распознает типы прикрепленных
  файлов через data.files (с обратной совместимостью со старым форматом
  data.fileUrl): картинки → <img>, видео → <video>, документы (PDF, ZIP,
  DOCX) → красивый виджет-плашка с оригинальным именем файла и кнопкой
  прямого скачивания с домашнего ПК. Все пользовательские данные проходят
  через escapeHtml() для защиты от XSS.

### Безопасность на сервере (server.js):
- Внедрен Rate Limiting (express-rate-limit) с лимитом 50 запросов
  за 2 минуты (увеличено с 5, т.к. мультизагрузка делает по одному
  POST-запросу на каждый файл).
- Жесткие лимиты Multer: 300 МБ для видео, 50 МБ для фото и документов
  (проверка размера через limits.fileSize и через content-length
  в fileFilter).
- Имена файлов очищаются регулярным выражением от опасных спецсимволов,
  обрезаются до 30 символов, к ним добавляется временная метка Date.now().
- Запрещены к загрузке исполняемые файлы (.exe, .sh, .bat) и любые типы
  не из белого списка (images/*, video/*, application/pdf, zip, doc(x),
  xls(x), ppt, text/*).
- Глобальный перехватчик ошибок Express + process.on('uncaughtException')
  и 'unhandledRejection' защищают Node.js от падения при обрыве
  соединения. Заголовки CORS прописаны жёстко (с поддержкой
  Bypass-Tunnel-Reminder для прохождения через localtunnel).
- Раздача статики /uploads через express.static вынесена ВЫШЕ rate-лимитера,
  чтобы отдача файлов не съедала квоту загрузок.

### Деплой:
- Фронтенд собирается через `npm run build` (Vite) и публикуется на
  GitHub Pages командой `npm run deploy` (пакет gh-pages, папка dist).
- В vite.config.js прописан base: '/my-chat/' для корректной работы
  путей на GitHub Pages.
- Бэкенд работает локально, проброшен через localtunnel. URL туннеля
  меняется при каждом перезапуске, поэтому его нужно обновлять в
  .env и пересобирать фронт.
