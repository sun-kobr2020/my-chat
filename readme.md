У меня есть веб-мессенджер (SPA) с комнатами, развернутый на GitHub Pages
по адресу https://sun-kobr2020.github.io/my-chat/ и собранный на Vite
(JS, HTML5, CSS3).

### Текущая гибридная архитектура проекта:
1. База данных и Real-time (Firebase v9+ Firestore): Текст сообщений, имена
   комнат и метаданные синхронизируются через Firebase в реальном времени
   (onSnapshot).
2. Собственное self-hosted хранилище (Node.js + Express + Multer): Все
   медиафайлы и документы загружаются и хранятся на моем домашнем сервере
   (ПК с 64 ГБ RAM), а не в Firebase Storage. Бэкенд проброшен в интернет
   через бесплатный туннель (на данный момент используется localtunnel:
   `npx localtunnel --port 5000 --print-requests`, планируется переход на
   Pinggy через `ssh -p 443 -R0:localhost:5000 a.pinggy.io` ради
   отсутствия страницы-предупреждения и поддержки крупных файлов до 1 ГБ+).

### Структура и файлы проекта:
1. index.html — Двухпанельный интерфейс. Слева панель комнат с кнопкой
   "Создать комнату" и списком доступных комнат. Справа панель чата
   с заголовком текущей комнаты, окном сообщений и блоком '.input-panel'
   внизу. Прямо над строкой ввода '.message-row' встроен служебный блок
   индикации файлов `#file-preview-zone` с кнопкой отмены
   `#cancel-file-btn`.
2. style.css — Стили сетки чата, комнат, плашек файлов и адаптивная
   верстка.
3. app.js — Главный скрипт (модульный SDK Firebase). Обрабатывает логику
   комнат, слушает Firestore через onSnapshot, управляет отправкой
   сообщений и мультизагрузкой файлов через fetch на удалённый бэкенд.
   Поддерживает escapeHtml() для защиты от XSS.
4. server.js — Бэкенд-сервер на Node.js (работает локально на порту 5000),
   принимающий файлы от фронтенда. Имеет двойной CORS-блок (cors() +
   ручные заголовки), включая обработку preflight OPTIONS-запросов и
   заголовка Bypass-Tunnel-Reminder. Защита от падения через
   process.on('uncaughtException') и 'unhandledRejection'.
5. .env / .env.production — Содержат ключи VITE_FIREBASE_* и переменную
   VITE_FILE_SERVER_URL (в режиме разработки → http://localhost:5000,
   в продакшене → актуальный URL туннеля). Папка /uploads/ и .env скрыты
   в .gitignore.
6. vite.config.js — содержит `base: '/my-chat/'` для корректной работы
   путей на GitHub Pages.
7. package.json — содержит скрипты:
    - "dev": "vite"
    - "build": "vite build"
    - "preview": "vite preview",
    - "deploy": "gh-pages -d dist"
    - "redeploy": "npm run build && npm run deploy"

### Логика работы с файлами и сообщениями:
- Фронтенд поддерживает мультизагрузку (multiple) любых типов файлов через
  функцию handleFileUpload(e). Файлы предварительно копируются в массив
  через Array.from(e.target.files), а очистка input.value происходит ТОЛЬКО
  в блоке finally (после полного завершения загрузки), что устранило баг
  с обнулением FileList в браузере.
- Файлы отправляются поочерёдно в цикле через FormData (поле 'chatFile')
  на бэкенд. fetch вызывается БЕЗ кастомных заголовков, чтобы запрос
  оставался "simple CORS" и не порождал preflight (иначе localtunnel
  блокирует OPTIONS).
- Бэкенд динамически сортирует файлы на жестком диске по папкам:
  /uploads/images/, /uploads/videos/ или /uploads/documents/ на основе
  MIME-типов через функцию getSubFolder(mimetype).
- Бэкенд возвращает объект {fileUrl, fileType} для каждого файла. Фронт
  собирает массив attachedFilesList. Серая плашка превью динамически
  меняет цвета и статусы («⏳ Отправка X из Y...» → «✅ Файлы на сервере!
  Готово: N шт.» → «🚀 Отправка в чат...»). В случае сбоя выводится
  ошибка красным.
- Функция sendMessage() упаковывает текст и массив файлов в один объект
  и отправляет в Firestore по пути: rooms / {currentRoomId} / messages.
  После публикации интерфейс и массив attachedFilesList полностью
  очищаются.
- Рендеринг внутри onSnapshot автоматически распознает типы прикрепленных
  файлов через data.files (с обратной совместимостью со старым форматом
  data.fileUrl): картинки → <img>, видео → <video>, документы (PDF, ZIP,
  DOCX, XLSX) → красивый виджет-плашка с оригинальным именем файла и
  кнопкой прямого скачивания с домашнего ПК. Все пользовательские данные
  проходят через escapeHtml() для защиты от XSS.

### Логика комнат:
- Создание новой комнаты — кнопка "Создать комнату" генерирует случайный
  ID `rm-XXXXXXXX`, регистрирует его в коллекции "all_rooms" Firestore
  и переключает window.location на новый ?room=XXX.
- Список комнат отображается слева через onSnapshot к "all_rooms",
  включая кнопку "❌" для удаления (deleteDoc).
- При currentRoom !== 'general' появляются кнопки "Поделиться ссылкой"
  (копирует URL в буфер) и "Вернуться в общий чат".

### Безопасность на сервере (server.js):
- Внедрен Rate Limiting (express-rate-limit) с лимитом 50 запросов
  за 2 минуты (увеличено с 5, т.к. мультизагрузка делает по одному
  POST-запросу на каждый файл).
- Жесткие лимиты Multer: 300 МБ через limits.fileSize (абсолютный
  потолок), 50 МБ для фото и документов через дополнительную проверку
  content-length в fileFilter.
- Имена файлов очищаются регулярным выражением от опасных спецсимволов,
  обрезаются до 30 символов, к ним добавляется временная метка Date.now().
- Запрещены к загрузке исполняемые файлы (.exe, .sh, .bat) и любые типы
  не из белого списка (images/*, video/*, application/pdf, zip, doc(x),
  xls(x), ppt(x), text/*).
- Раздача статики /uploads через express.static вынесена ВЫШЕ
  rate-лимитера, чтобы отдача файлов не съедала квоту загрузок.
- Глобальный перехватчик ошибок Express + process.on() защищают Node.js
  от падения при обрыве соединения. CORS прописан с поддержкой заголовка
  Bypass-Tunnel-Reminder для прохождения через туннели.

### Деплой:
- Фронтенд собирается через `npm run build` (Vite) и публикуется на
  GitHub Pages командой `npm run deploy` (пакет gh-pages, папка dist).
- Бэкенд работает локально на ПК, проброшен через туннель. URL туннеля
  меняется при каждом перезапуске, поэтому его нужно обновлять в .env
  и пересобирать фронт через `npm run redeploy`.
- Правильная последовательность запуска для теста:
    1. `node server.js` (терминал 1)
    2. `npx localtunnel --port 5000 --print-requests` (терминал 2)
    3. Скопировать URL → обновить .env
    4. Открыть URL туннеля в браузере и пройти страницу Continue
    5. Для прода: `npm run redeploy`

### Известные ограничения и причины перехода с localtunnel на Pinggy:
- localtunnel показывает страницу-предупреждение (каждый пользователь
  должен один раз пройти Continue).
- localtunnel ломает CORS preflight (OPTIONS-запросы блокируются туннелем
  до достижения сервера).
- localtunnel нестабилен, рвёт соединение через несколько минут.
- Cloudflare Tunnel не подошёл из-за лимита 100 МБ на размер запроса.
- Pinggy → нет страницы-предупреждения, корректный CORS, без лимитов
  размера, но бесплатная сессия 60 минут.

### Текущий контекст сетевой инфраструктуры проекта (Перешли на Pinggy):
1. Текущий туннель: Для вывода домашнего файлового сервера в интернет используется инструмент туннелирования Pinggy.io, работающий через встроенный SSH-клиент Windows.
2. Команда запуска туннеля:
   ssh -p 443 -o StrictHostKeyChecking=no -o ServerAliveInterval=30 -R 80:127.0.0.1:5000 guest@a.pinggy.io
   (Туннель связывает внешний публичный HTTPS-адрес с локальным портом бэкенда 5000).
3. Настройки во фронтенде (.env): В переменной окружения `VITE_FILE_SERVER_URL` зафиксирован текущий временный адрес, выданный Pinggy (формата https://pinggy-free.link). Хвостик приложения /my-chat/ из URL-пути удален. При перезапуске туннеля адрес обновляется вручную только в файле .env с последующей очисткой кэша сборщика через `npm run dev -- --force`.
4. Настройки на бэкенде (server.js):
    - Сервер Node.js/Express жестко привязан к прослушиванию IPv4-интерфейсов через `app.listen(PORT, '0.0.0.0', ...)`, чтобы входящий из туннеля трафик по адресу 127.0.0.1:5000 гарантированно находил бэкенд и не вызывал ошибку 503 Service Unavailable.
    - Для обхода специфического поведения прокси-серверов Pinggy, которые могут срезать заголовки ответов, в `server.js` встроен жесткий перехватчик CORS. Он принудительно внедряет заголовки Access-Control-Allow-* во все типы ответов и моментально возвращает статус 200 OK на предварительные проверочные запросы браузера (preflight OPTIONS), предотвращая блокировки со стороны CORS policy.
5. Лимиты на файлы через туннель: Использование Pinggy позволило полностью обойти жесткое ограничение Cloudflare в 100 МБ. В конфигурации Multer на сервере выставлены новые расширенные лимиты: 300 МБ для видео и по 50 МБ для фотографий и документов.
